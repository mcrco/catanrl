# Overview

This was a project for me to learn RL by training a bot to play Catan. I've implemented a few different algorithms:

- supervised learning on datasets generated by bot play (has been removed since it performed terribly)
- imitation learning with DAgger
- single agent PPO
- multi agent self-play PPO with centralized critic
- AlphaZero style training

Disclaimer: I initially implemented SL and the PPO variants by hand, but over time, that code has undergone a Ship of Theseus series of modifications since I started using more AI.

## Project Structure

The project is a `uv`-managed Python package (`catanrl`) that builds on a custom fork of a vendored
[Catanatron](https://github.com/bcollazo/catanatron) fork (in `catanatron/`, tracked as a
submodule) for the game engine and baseline opponents.

```
src/catanrl/
  algorithms/          # the actual learning algorithms
    imitation_learning/  # DAgger trainer + dataset
    ppo/                 # single-agent PPO and multi-agent central-critic PPO
    alphazero/           # AlphaZero-style trainer + parallel self-play
    common/              # shared bits (e.g. action selection)
  experiments/         # training entrypoints (one per algorithm) + arg/config plumbing
  experiment_store.py  # save/load self-describing experiments (see below)
  models/              # network components: backbones, heads, builders, wrappers
  envs/puffer/         # PufferLib env wrappers (single- and multi-agent), rewards, rollouts
  features/            # turn Catanatron game states into model observations
  beliefs/             # opponent hidden-info modeling (dev-card belief, determinization)
  players/             # bots that plug into Catanatron (policy, value, MCTS, alpha-beta)
  eval/                # evaluation harnesses (vs Catanatron, during-training eval)
  utils/               # Catanatron action-space/map/game helpers, seeding
  profiling/           # performance profiling scripts

configs/models/        # YAML architecture/game presets used to start training runs
scripts/               # standalone eval/profiling scripts
tests/                 # pytest suite
experiments/           # (gitignored) saved training runs, see below
```

Training entrypoints are exposed as console scripts (see `pyproject.toml`):

- `train-dagger` — imitation learning (DAgger)
- `train-sarl-ppo` — single-agent PPO
- `train-marl-cc` — multi-agent self-play PPO with a centralized critic
- `train-alphazero` — AlphaZero-style training

### Experiments

Every training run is a **self-describing experiment** under `experiments/<name>/`
(gitignored). The directory holds everything needed to reload a model later:

```
experiments/<name>/
  metadata.json      # exact architecture (backbone + head type) of each network,
                     #   plus the game config it was trained on (map_type,
                     #   num_players, vps_to_win, discard_limit)
  checkpoints.json   # best/latest/step -> checkpoint file, per role (policy/critic)
  checkpoints/       # the .pt state_dicts
```

Because `metadata.json` records the precise architecture, you never have to
re-specify model settings at eval time — the model rebuilds itself from the experiment.

#### Training

Training scripts require **either** `--config` (YAML preset under `configs/models/`)
**or** `--load-from-experiment` (warm start; architecture comes from experiment
metadata). If both are passed, the architectures must match exactly.

Also pass `--experiment-name` (it becomes the W&B run name when set). If omitted,
it falls back to `--wandb-run-name`, else an auto-generated `<algo>-<timestamp>`.

```
uv run train-sarl-ppo \
  --config configs/models/xdim-flat-2p-private.yaml \
  --experiment-name ppo-sarl-f-winreward-xdim-flat-pretrained-dagger \
  --opponents F --wandb
```

Architecture presets define the model backbone, observation levels, and game
settings (`map_type`, `vps_to_win`, etc.). The full resolved architecture is
saved into experiment metadata when training finishes — not the YAML path.

#### Loading a model in code

```python
from catanrl.experiment_store import load_experiment

exp = load_experiment("ppo-sarl-f-winreward-xdim-flat-pretrained-dagger")
policy = exp.build_policy(which="best", device="cuda")   # arch rebuilt + weights loaded
critic = exp.build_critic(which="best", device="cuda")   # if the run has a critic
```

`which` accepts `"best"`, `"latest"`, or an explicit training step.

#### Evaluating

Pass `--experiment` (architecture is read from metadata); `--which` selects the
checkpoint. The legacy `--policy-weights` + architecture flags still work.

```
uv run scripts/eval_vs_catanatron.py \
  --experiment ppo-sarl-f-winreward-xdim-flat-pretrained-dagger \
  --which best \
  --num-games 1000 \
  --seed 67 \
  --nn-seat {first/second} \
  --opponents {AB:2/F}
```

#### Managing experiments

Experiments are written automatically by the training entrypoints under
`src/catanrl/experiments/`. Each run is self-describing: list them with
`ls experiments/`, and inspect any run by reading its plain-JSON
`experiments/<name>/metadata.json` and `experiments/<name>/checkpoints.json`.
Because each run records its exact network architecture in `metadata.json`
directly from the trained model, `load_experiment(...)` rebuilds loadable models
without re-deriving anything from the weights.

Plan: lock in `xdim` model size on DAgger with fast iterations, confirm shared
backbone ≈ separate backbone, then on SARL/MARL settle shared-vs-separate and
full-vs-public info (no private — no reason to train it), and finally take the
winning arch (expected: shared backbone, ~2x faster) into long AlphaZero runs.

All win rates are vs the listed opponent over 1000 games at a fixed seed,
`first / second` seat (`scripts/eval_vs_catanatron.py --experiment <name> --which best`). Status is `WIP` until a run is done, then fill in the numbers.

### MARL self-play (centralized critic)

MARL central-critic now supports `shared` (only when actor/critic info levels
match, e.g. `public/public` or `full/full`). Two real-game (public-info)
deployment recipes are compared:

- **public, direct** — train `public/public`, deploy the actor straight on real
public info (`NNPolicyPlayer`), no belief needed.
- **full, belief** — train `full/full`, deploy via `BeliefAveragedPolicyPlayer`
(marginalize the full-info policy over the dev-card belief at test time).


| ID            | network  | policy/critic mode | deploy / eval | Status | Win% vs F (1st/2nd) | Win% vs AB | Notes                             |
| ------------- | -------- | ------------------ | ------------- | ------ | ------------------- | ---------- | --------------------------------- |
| M-sep-pub     | separate | public / public    | direct        | WIP    |                     |            |                                   |
| M-shared-pub  | shared   | public / public    | direct        | WIP    |                     |            | ≈2x faster                        |
| M-sep-full    | separate | full / full        | belief-avg    | WIP    |                     |            |                                   |
| M-shared-full | shared   | full / full        | belief-avg    | WIP    |                     |            | ≈2x faster                        |
| M-sep-priv    | separate | public / full      | direct        | WIP    |                     |            | privileged critic (separate only) |


➡ Two questions to settle here: (1) shared ≈ separate? if so, lock in shared
(≈2x faster); (2) which real-game recipe wins — public-direct vs full-belief?
Carry the winning arch + recipe into AlphaZero.

## Phase 2.5 — Eval experiments (test-time only, no extra training)

Two test-time studies on the trained MARL agents. Both gate/inform Phase 3 and
use the eval scripts directly — no new training runs.

### MARL agent inside the AlphaZero/MCTS harness (before AlphaZero training)

Wrap a trained MARL policy+critic in `NNMCTSPlayer` and search at test time
(`scripts/eval_mcts_vs_catanatron.py --experiment <marl-exp> --which best`).
Question: does test-time MCTS on top of the MARL net buy win rate, and how much —
i.e. is AlphaZero training worth it, and what gain should we expect? Compare
against the same agent with no search (raw `NNPolicyPlayer`) as the baseline.


| ID         | base MARL agent  | num-simulations | ismcts-determ. | Status | Win% vs F (1st/2nd) | Win% vs AB | Notes                      |
| ---------- | ---------------- | --------------- | -------------- | ------ | ------------------- | ---------- | -------------------------- |
| H-raw      | best MARL        | — (no search)   | —              | WIP    |                     |            | baseline (policy only)     |
| H-mcts-64  | best MARL        | 64              | 1              | WIP    |                     |            | plain MCTS                 |
| H-mcts-128 | best MARL        | 128             | 1              | WIP    |                     |            | more search                |
| H-ismcts   | best MARL (full) | 64              | 8              | WIP    |                     |            | IS-MCTS (public/full only) |


➡ If search lifts win rate meaningfully, proceed to Phase 3; the lift sets the
expectation for what AlphaZero training should beat.

### Belief-averaging: "train on perfect info, average over states at test time"

The belief player **always wraps a `full/full` agent** — it marginalizes a
perfect-info policy over the opponent dev-card belief, so a `public`/`private`
net (which never sees full info) would make no sense here. Deploy it with
`BeliefAveragedPolicyPlayer` (`scripts/eval_belief_policy.py --experiment <full-marl-exp> --also-oracle`); the belief is exact in 1v1. Compare three points
to quantify how much hidden info matters to the net:

- **oracle** — full-info net seeing true hidden cards (cheats; upper bound).
- **belief-averaged** — same net, no cheating, averaged over the belief.
- **public native** — the natively `public/public` MARL agent (reference).

If belief-averaged ≈ public native, then training on perfect info + averaging is
a viable substitute; the oracle − belief gap measures how much private info is
worth to the bot.


| ID       | agent (train info) | deploy / eval             | Status | Win% vs F (1st/2nd) | Win% vs AB | Notes       |
| -------- | ------------------ | ------------------------- | ------ | ------------------- | ---------- | ----------- |
| B-oracle | full / full        | full-info oracle (cheats) | WIP    |                     |            | upper bound |
| B-belief | full / full        | belief-averaged           | WIP    |                     |            | no cheating |
| B-public | public / public    | direct (`NNPolicyPlayer`) | WIP    |                     |            | reference   |


➡ Records: belief vs public answers "is perfect-info + averaging comparable?";
oracle vs belief answers "how much does private info matter?".

## Model Architecture

- CNN features for board,
- MLP for numeric features (e.g. player hands)
- then concat them and do one more MLP on the fused features
- basically a super washed down version of [this paper](https://arxiv.org/abs/2008.07079).
- see `src/catanrl/models`

## Training Algorithms

- first do imitation learning (DAgger) on the Catanatron `ValueFunctionPlayer` for a decent base model
- then do PPO against the `ValueFunctionPlayer`

## Winrate

- 70.6% over 1000 games against ValueFunction when going first
- 67.7% over 1000 games against ValueFunction when going second
- 69.7% over 1000 games against AlphaBeta when going first
- 66.0% over 1000 games against AlphaBeta when going second
